# PynqCコンパイラ構成

## 処理パイプライン

`Source → Lexer → Token → Parser → AST → Semantic → 型付きAST → Codegen → .s`の順で処理する。
外部Parser Generatorや外部クレートへ依存せず、入力エラーは`Diagnostic`で返す。

- `source.rs`: byte offset、行、列、ソース行を管理する。
- `lexer.rs` / `token.rs`: コメント、リテラル、演算子をトークン化する。
- `parser.rs` / `ast.rs`: 再帰下降と優先順位上昇法でSpan付きASTを構築する。
- `types.rs`: 型、サイズ、アラインメント、暗黙変換を定義する。
- `symbol.rs` / `semantic.rs`: 二段階でグローバル名を収集し、スコープと型を検査する。
- `frame.rs`: 全ローカルを固定スタックオフセットへ配置する。
- `codegen.rs`: 可読な既存ISAアセンブリと最小ランタイムを生成する。
- `diagnostic.rs`: エラー分類、ソース行、キャレット、補足を表示する。

## ASTと意味解析

ASTの各節点は`Span`を持つ。意味解析は変数参照へ一意な`SymbolId`、式へ`Type`と左辺値属性を
付加する。関数シグネチャは本体解析前に全件収集するため前方呼出しと再帰が可能である。
スコープはHashMapのスタックで表し、内側から名前を検索する。

型検査は未定義名、重複、引数、return、void値、左辺値、配列代入、ポインタ演算、ループ外
break/continueを拒否する。完全な到達可能性解析は行わず、ifの両枝または後続の明示returnを
基準に保守的に判定する。

## コード生成

左辺値アドレス生成と右辺値ロードを`emit_lvalue`と`emit_expr`で分離する。式結果は`r6`、
二項演算の左辺は評価スタックから`r7`へ戻す。深い式はCPUスタックへ退避するので固定個数の
一時レジスタに依存しない。関数呼出し前の式値もスタック上にあるためcaller-saved破壊を受けない。

比較は直接比較分岐と一意な`.L_*`ラベルで0/1へ正規化する。`&&`と`||`は分岐で短絡する。
ポインタ演算は要素サイズ1または4を加算前にスケールする。乗除算はランタイム呼出しへ変換する。

## データ配置

命令ROMとデータRAMが分離され、CPUはROMをデータとして読めない。このため`.rodata/.data`
をROMに置かず、グローバルと文字列へ`0x4000`からアドレスを割り当て、`_start`がSTOREB/STOREで
初期化する。コードは単一アセンブリなので再配置とリンカを必要としない。この方式は単純だが
初期化コードが増えるため、将来はデータRAM初期化イメージまたはブートローダを導入する。

## 今後のIR

最適化開始時には型付きASTとISAコード生成の間へ、基本ブロック、仮想レジスタ、明示的load/store、
callを持つ小さなSSA前IRを追加する。現段階では直接生成の方がABI対応を追跡しやすい。
